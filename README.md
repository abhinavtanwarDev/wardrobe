# PWA App

A production-ready Progressive Web App scaffold built with Next.js 16 (App
Router), TypeScript, Tailwind CSS v4, and `@serwist/next`.

## Setup

```bash
npm install
cp .env.example .env.local
npm run dev
```

Open http://localhost:3000. The service worker is disabled in development
(`next dev`) to avoid stale-cache confusion while iterating — test PWA
behavior with a production build instead:

```bash
npm run build
npm run start
```

## Scripts

| Script                   | Command                           | Purpose                                          |
| ------------------------ | --------------------------------- | ------------------------------------------------ |
| `npm run dev`            | `next dev --webpack`              | Start the dev server (no service worker)         |
| `npm run build`          | `next build --webpack`            | Production build (generates `public/sw.js`)      |
| `npm run start`          | `next start`                      | Serve the production build                       |
| `npm run lint`           | `eslint .`                        | ESLint                                           |
| `npm run format`         | `prettier --write .`              | Prettier, writes fixes                           |
| `npm run format:check`   | `prettier --check .`              | Prettier, check only                             |
| `npm run typecheck`      | `tsc --noEmit`                    | Type checking                                    |
| `npm run generate-icons` | `node scripts/generate-icons.mjs` | Regenerate `public/icons/*` from the master SVGs |

### Why `--webpack`?

`dev` and `build` pass `--webpack` explicitly. Next.js 16 defaults to
Turbopack, but `@serwist/next` (the service-worker build plugin) injects a
webpack config into `next.config.mjs` to compile `src/app/sw.ts` into
`public/sw.js`, and Turbopack hard-errors when it detects a webpack config
present. Passing `--webpack` is the documented Serwist workaround for
Next.js 16. **Do not remove this flag** — doing so breaks both `dev` and
`build`. If a future Serwist/Next.js release adds Turbopack support for
custom service-worker compilation, this flag can be dropped.

## Folder structure

```
src/
  app/            # routes, layouts, metadata (App Router)
    api/items/    # sample API route used by the offline demo
    offline/      # /offline fallback page shown when a navigation fails
    sw.ts         # service worker source, compiled to public/sw.js
  components/
    ui/           # reusable primitives (Button, Card)
    layout/       # Header, Footer, OfflineBanner, InstallPrompt, UpdateToast
  features/       # feature-sliced modules (e.g. features/items)
  hooks/          # useOnlineStatus, useInstallPrompt
  lib/            # constants, utilities
  styles/         # globals.css, tokens.css
  types/          # shared TypeScript types
public/
  icons/          # generated PWA icon set + master SVGs
  manifest.webmanifest
  sw.js           # build output, generated by @serwist/next — do not edit
scripts/
  generate-icons.mjs
```

## How the service worker and caching work

`@serwist/next` wraps the Next.js build to compile `src/app/sw.ts` into
`public/sw.js`. Caching strategies (defined in `src/app/sw.ts`):

- **`/api/*` routes** — `NetworkFirst`: try the network, fall back to the
  last cached response (named cache `api-cache`, entries expire after 24h
  or once there are more than 32, via `ExpirationPlugin`).
- **Everything else** (pages, static assets, images) — Serwist's
  `defaultCache`, which applies `StaleWhileRevalidate`/`CacheFirst`
  depending on asset type.
- **Failed document navigations** (e.g. offline + no cached page) — served
  the `/offline` fallback page, matched via `fallbacks.entries` on
  `request.destination === "document"`.
- **Updates**: the service worker is registered with `skipWaiting: false`
  and `clientsClaim: true`, so a new version sits "waiting" until the user
  confirms via the "New version available — Refresh" toast (`UpdateToast`
  component), which posts a `SKIP_WAITING` message to the waiting worker;
  the worker calls `self.skipWaiting()` on that message and the page
  reloads once the new worker takes control.
- `navigationPreload: true` is also enabled to speed up the `NetworkFirst`
  strategy's network attempt on supporting browsers.

## Testing offline mode

1. `npm run build && npm run start`
2. Open the app in Chrome, open DevTools → Application → Service Workers,
   confirm the worker is "activated and is running".
3. DevTools → Network → set throttling to "Offline".
4. Reload — the last-cached page (or `/offline` if nothing was cached yet)
   should still render, and the amber "You're offline" banner should
   appear.
5. Go back online and reload — the banner disappears and fresh data loads.

## Testing installation

- **Desktop Chrome/Edge**: an install icon appears in the address bar once
  the manifest + service worker are detected; the in-page "Install" banner
  (`InstallPrompt`) also triggers the same native prompt.
- **Android Chrome**: the browser's "Add to Home screen" banner appears
  automatically, or use the in-page prompt.
- **iOS Safari**: `beforeinstallprompt` doesn't exist on iOS, so the
  `InstallPrompt` component instead shows manual instructions ("tap Share,
  then Add to Home Screen") when it detects Safari on iOS and the app
  isn't already running standalone.

## Customizing the manifest and icons

- Edit `public/manifest.webmanifest` for `name`, `short_name`,
  `description`, `theme_color`, `background_color`.
- Replace `public/icons/icon.svg` (standard icon) and
  `public/icons/icon-maskable.svg` (full-bleed background, keep the main
  glyph inside the inner ~80% — the rest gets cropped by OS mask shapes)
  with your own branding, then run `npm run generate-icons` to regenerate
  every PNG size.
- Update `THEME_COLOR`/`BACKGROUND_COLOR`/`APP_NAME`/`APP_DESCRIPTION` in
  `src/lib/constants.ts` to keep the manifest and the `<head>` metadata in
  `src/app/layout.tsx` in sync.

## Verifying with Lighthouse

```bash
npm run build && npm run start
npx lighthouse http://localhost:3000 --view --preset=desktop
```

Check the "Installable" and "PWA optimized" sections of the report:
manifest validity, icon coverage, service worker registration, and offline
start_url response are all audited automatically. Chrome DevTools →
Lighthouse tab works the same way without the CLI.

## Skills followed

This project was scaffolded using guidance from the installed `nextjs`
and `progressive-web-app` skills (App Router conventions, manifest/service
worker/install-prompt examples). Both skills provide generic reference
patterns rather than prescribing specific libraries or pinning to a single
framework version, so the following were decided during implementation and
documented in `docs/superpowers/specs/2026-07-06-pwa-scaffold-design.md`:

- Library choices: `@serwist/next` for the service worker, Tailwind CSS v4,
  `sharp` for icon generation.
- Next.js 16 version-drift adaptations: the `--webpack` flag on `dev` and
  `build` (see [Why `--webpack`?](#why---webpack) above), and using
  `eslint-config-next`'s native flat-config export directly in
  `eslint.config.mjs` instead of the legacy `.eslintrc` + `FlatCompat`
  pattern.
